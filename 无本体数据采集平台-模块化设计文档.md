# 无本体数据采集平台 — 模块化设计文档

> 面向 Embodied AI 策略训练的数据采集与处理平台。
> 核心目标:**高质量、可追溯、可复现、可直接对接训练**。

---

## 0. 设计原则

1. **一切可追溯**:从 raw 到训练集,每一步处理都能回溯参数和数据血缘。
2. **采集即对齐**:时间戳、坐标系在采集端就统一,不依赖后处理兜底。
3. **采集 ≡ 部署**:采集配置(相机位姿、FOV、topic 顺序)必须能机器化校验,与部署一致。
4. **原始数据只读**:所有清洗、对齐、重定向的产物都是派生数据,raw 永不覆盖。
5. **质量优先于数量**:宁可少采,不要脏数据污染策略。

## ① 采集前置

### 1.1 多设备时间同步
- **目标**:所有传感器 / 执行器数据共享统一时间基准,采集端打时间戳,而非事后对齐。
- **选型档位**:
  - 硬件触发(最高精度,相机/IMU 同帧触发)
  - PTP (IEEE 1588) / gPTP(网络同步,亚毫秒级)
  - NTP / Chrony(软同步,毫秒级,作为最低保障)
- **交付物**:同步方案说明、各设备时间源配置、同步偏差基线报告(周期性自检)。
- **验收指标**:任意两设备时间戳偏差 < 某阈值(按任务敏感度定)。

### 1.2 标定与坐标系统一
- **标定对象**:相机内参、多相机外参、手/臂基座变换、传感器零位。
- **关键约束**:
  - 标定结果 + 标定原始数据都存档,可追溯重算。
  - 每个 session 必须绑定一个标定版本号。
  - 场景坐标系原点定义明确(桌面某角 / 机械臂 base 等)。
- **交付物**:标定工具、标定产物版本库、session ↔ 标定版本映射表。

### 1.3 任务定义 (Task Schema)
结构化任务描述,避免塞在文件名或口头约定里。
```yaml
task_id: pick_place_block_001
language_instruction: "把红色方块放到蓝色盘子里"
objects:
  - {name: red_block, type: cube, init_pose_range: [...]}
  - {name: blue_plate, type: plate, init_pose_range: [...]}
success_criteria:
  - red_block.position in blue_plate.bbox
  - gripper_released == true
max_duration_s: 30
variation_axes: [object_pose, lighting, distractor_count]
```

## ② 数据采集

### 2.1 设备接入与数据流
- 统一设备抽象层(Driver SDK / ROS2 node / ZMQ topic 等任选其一并全平台一致)。
- 每路传感器配置:采样率、时间戳来源、数据类型、编码格式。
- 录制格式推荐 **rosbag2 / MCAP / HDF5 shard**,避免单一巨型文件。

### 2.2 操作员管理
- operator ID、资质等级(新手 / 熟练 / 专家)、授权任务范围。
- 每条 episode 绑定 operator ID,用于:
  - 数据分布分析(某 operator 是否偏态)
  - 质量分层(优先使用专家数据)
  - 分布漂移排查(policy 效果变差是数据变了,还是操作员换了?)

### 2.3 实时质量监控
**采集当下就告警,而不是回来才发现**。
- 监控项:丢帧率、传感器延迟、设备掉线、数值跳变、力/速度超限。
- UI:采集界面实时仪表盘(红黄绿信号灯)。
- 策略:严重问题自动暂停采集,提示重采。

### 2.4 SOP 与重采机制
- 标准开始流程:自检 → 标定校验 → 任务载入 → 起始姿态检查。
- 标准结束流程:成功判定 → 数据完整性校验 → 元数据落盘。
- 废弃 / 重采标记:操作员可一键标记"失败,丢弃"或"失败,保留作为 negative"。

## ③ 数据格式

### 3.1 原始数据 (raw)
- 原生格式保留(rosbag2 / MCAP / 各设备原始封装)。
- **原则:原始数据只读,任何后处理不得覆盖 raw**。

### 3.2 元数据 Schema
每条 episode 附带结构化 metadata,字段至少包含:
```yaml
episode_id: <uuid>
task_id: <see Task Schema>
operator_id: <id>
timestamp_start: <ISO8601>
timestamp_end: <ISO8601>
devices: [<device_id list>]
calibration_version: <id>
sop_version: <id>
success_label: success | partial | fail | discard
quality_flags: [low_fps_cam1, imu_jump, ...]
notes: <free text>
```

### 3.3 标准化格式(面向训练)
- 输出标准:**LeRobotDataset / RLDS / HDF5**(按下游 policy 栈选定)。
- 每个标准格式都附 schema 版本号。

---

## ④ 数据上传

### 4.1 上传协议
- 分块上传 + 校验(SHA256 per chunk)。
- 断点续传。
- 优先压缩传输(视频流用 H.264/H.265,状态流用 zstd)。

### 4.2 分级存储
- 热存储:近 30 天 / 活跃训练数据。
- 温存储:归档但可查。
- 冷存储:超过 N 月自动下沉(raw 永远保留,派生数据可重算)。

### 4.3 目录规范
```
/<task_id>/<operator_id>/<yyyymmdd>/<episode_id>/
    raw/
    metadata.yaml
    calibration_ref.yaml
```

## ⑤ 数据后处理

### 5.1 数据清洗
- 丢帧 / 跳变检测与修补(线性插值 vs 丢弃 episode,按字段策略区分)。
- 时间戳单调性修正。
- 坐标系统一到 session 绑定的标定坐标系。
- 无效 episode 自动过滤(时长过短、传感器缺失、成功标记为 discard)。

### 5.2 多模态 & 多设备精准对齐
- 以主时钟为基准,重采样或插值到统一频率。
- 不同模态对齐策略分别定义(图像用最近邻,姿态用插值,离散事件保留原时刻)。
- 输出对齐报告:各模态 vs 主时钟的最大偏差。

### 5.3 数据重定向 (Retargeting)
- 源空间 → 目标空间(例:人手姿态 → 机械手关节空间)。
- 重定向算法版本化,每条数据记录重定向版本号。
- **必须配合 §7.1 回放一致性测试验证质量**。

### 5.4 数据格式转换
- raw → 训练格式的转换流水线。
- 幂等、可重跑、参数版本化。

## ⑥ 数据标注

### 6.1 Episode 切分与关键帧
- 自动切分(基于动作变化 / 接触事件 / 夹爪开合) + 人工校正。
- 关键帧标注用于分层策略 / skill chaining。

### 6.2 语言指令 / 子任务标签
- 高层语言指令(整个 episode)。
- 子任务分段标签(approach → grasp → transport → place)。
- 支持多语言指令的同义重写(数据增广)。

### 6.3 成败与分级标签
- success / partial_success / fail / discard。
- 失败数据**保留**,对某些训练范式(IRL、负样本学习)有价值。
- 质量等级(A/B/C),对应操作员水平和轨迹平滑度。

## ⑦ 数据质量评估

> 这一块独立于"清洗",重点是**量化质量**,指导采集策略。

### 7.1 回放一致性测试 ★ 核心指标
- 将重定向后的轨迹在真机 / 仿真中回放 N 次。
- 度量回放轨迹与原轨迹的误差(末端位置 / 物体最终位姿 / 成功率)。
- 回放一致性差 = 采集或重定向有问题,直接指导改进。

### 7.2 分布覆盖度分析
- 物体位姿覆盖(热力图)。
- 动作空间覆盖(关节 / 末端速度直方图)。
- 接触事件 / 抓取姿态分布。
- 输出"薄弱区域"报告,指导下一轮补采。

### 7.3 异常检测
- 轨迹跳点、穿模、不合理力值、速度突变。
- 基于规则 + 统计离群值双重检测。

---

## ⑧ 数据管理与检索

### 8.1 版本管理
- 数据集整体版本化(推荐 DVC / LeRobot hub / 自建 git-lfs 风格方案)。
- 后处理算法升级不污染历史版本。

### 8.2 元数据索引
可按以下维度查询:
- task_id / operator_id / 时间段
- success 状态 / quality 等级
- 设备配置 / 标定版本
- 分布属性(物体位姿区间、轨迹长度)

### 8.3 数据血缘 (Lineage) ★ 关键
每条处理后数据可回溯:
```
training_dataset_v3.hdf5
   ← retarget_v2 (commit abc123, params: ...)
   ← align_v1   (commit def456, params: ...)
   ← clean_v1
   ← raw episode_id=xxx (operator=roy, calib=v7)
```

---

## ⑨ 训练对接

### 9.1 数据集打包
- 输出标准训练格式(LeRobotDataset / RLDS / HDF5 shard)。
- 元数据随数据集打包,训练端可直接读取。

### 9.2 划分策略
**不要随机切分,容易泄漏**。建议:
- 按 episode 切(同一 episode 不跨 split)。
- 按 operator 切(验证集用不同 operator,测泛化)。
- 按物体 / 场景切(测对未见条件的泛化)。
- 明确记录 split 方案到数据集元数据。

### 9.3 采集 ↔ 部署一致性校验 ★
机器可校验项:
- 相机 topic 顺序、分辨率、FOV、外参。
- 观测空间 / 动作空间维度与类型。
- 控制频率。

任何不一致 → 训练前自动拦截,避免 policy 在部署时因输入分布不一致失败。

---

## ⑩ 可视化与 Debug(横向模块)

- **多视角同步回放**:所有相机 + 状态曲线同步 scrub。
- **3D 轨迹叠加**:末端 / 物体 / 关节轨迹在 3D 场景中可视化。
- **状态面板**:关节、力、速度、夹爪开合曲线。
- **对比模式**:采集 vs 回放 vs policy 推理结果三路对齐。

这个模块不是锦上添花,是**每天都会用**的,早做早省时间。

---

## ⑪ 隐私与合规(横向模块)

- 人脸 / 环境中敏感信息(工牌、屏幕内容)脱敏。
- 采集同意书管理(涉及人体动作数据时)。
- 数据授权范围与使用记录(内部 / 对外开源 / 合作方共享)。

---

## ⑫ 仿真联动(可选)

- **真机 → 仿真回放**:验证数据在仿真中能否复现,作为数据质量的附加门槛。
- **仿真数据增广反哺真机**:在仿真中批量生成物体位姿变化数据,缓解真机采集薄弱区域。
- **Sim-to-Real gap 量化**:记录同一轨迹在真 / 仿两侧的差异,作为仿真保真度指标。

## 实施优先级建议

按"最容易被跳过、后期补代价最大"排序,推荐分 3 期:

### P0(没有这些整个平台站不住)
1. **多设备时间同步** (§1.1)
2. **标定与坐标系存档** (§1.2)
3. **任务 Schema** (§1.3)
4. **元数据 Schema** (§3.2)
5. **采集 ↔ 部署一致性校验** (§9.3)

### P1(直接影响数据质量与训练效果)
6. 实时质量监控 (§2.3)
7. 回放一致性测试 (§7.1)
8. 多模态对齐 + 对齐报告 (§5.2)
9. 数据血缘 (§8.3)
10. 可视化回放 (§10)

### P2(规模化后必需)
11. 分布覆盖度分析 (§7.2)
12. 版本管理 / 元数据索引 (§8.1–8.2)
13. 操作员管理与数据分层 (§2.2)
14. 隐私合规 (§11)
15. 仿真联动 (§12)

## 附录 B:术语表

- **Episode**:一次完整任务执行的数据片段。
- **Session**:一批共享标定与配置的连续采集活动。
- **Retargeting**:将源空间动作映射到目标执行器空间。
- **Lineage**:数据血缘,从派生数据回溯到原始来源与处理链路。
- **Task Schema**:结构化的任务定义描述。
